# PoC：编辑助手直连 — 完整逆向工程记录

> 日期：2026-05-15
> 目标：验证不依赖 BI 编辑助手 SDK 也能从 Agent 端**端到端建出看板**
> 结果：✅ 完整闭环跑通（输入自然语言 → 自动建好可见的看板）
> 适用范围：仅限内部 PoC / 单用户场景；不投生产 / 不开放给真实用户

---

## TL;DR

**问题**：当前 Agent 输出"看板搭建指令"后，用户必须手动复制粘贴到 BI 编辑助手才能建好看板，存在流程断点。
**目标**：能不能跳过编辑助手 UI，直接调它的 API？
**结论**：能。但要过 4 道槛——TLS 指纹模拟、SSE 长连接、反向 RPC、双层持久化。
**完整端到端调用链**：8 个 endpoint，全部跑通，刷新 BI 真能看到 AI 建的图表。

## 一、最终调用链路

```
[Step 1] GET  /api/bigbi/os/sse/connect.do
              ?clientId=X&clientVersion=1.0.0&windowId=X
              → 建立 SSE 长连接
              ← event: connection.established { connectionId }

[Step 2] POST /api/bigbi/os/agent/messages/send.do
              ?projectId&osDataId&osMenuId
              body: { input.content (自然语言指令), context, clientId, conversationId: null }
              → 提交指令给 AI
              ← { conversationId, messageId }

[Step 3] SSE  ← event: query.page.json   ⭐反向 RPC 触发
[Step 4] POST /api/bigbi/os/agent/save/pageJson.do
              body: { pageJson (当前页面状态), conversationId, traceId, projectId }
              → 把当前页面状态发给服务端供 AI 参考

[Step 5] (AI 处理 ~30s)
[Step 6] SSE  ← event: push.message phase=completed { messageId, message }

[Step 7] POST /api/bigbi/os/agent/query/chunkData.do
              body: { conversationId, messageId, traceId, projectId }
              → 拉取 AI 生成的看板配置
              ← { result: "<stringified pageJson with new componentList>" }

[Step 8] POST /api/bigbi/os/agent/page/save.do (position=before)
              → AI 上下文档案：撤销点

[Step 9] POST /api/bigbi/os/agent/page/save.do (position=after)
              → AI 上下文档案：变更后状态（仅在 AI 对话里有效，不持久化）

[Step 10] ⭐ POST /api/bigbi/os/page/updatePageDraft.do
              body: { menuConfVO, pageSetting, componentList, cardList, ... } (顶层铺平,非字符串)
              → 真正写入看板 draft 存储
              ← { result: "<新 version 号>", success: true }
```

**关键观察**：
- Step 8/9 的 `agent/page/save.do` 看似在保存看板，**实际只是 AI 对话上下文的临时档案**——返回 200 但不写盘
- 真正写盘的是 Step 10 的 `page/updatePageDraft.do`，**命名空间从 `/agent/` 切到 `/page/`**
- response 的 `result` 字段是 version 号，能用于乐观锁

## 二、绕过的 4 道关卡

### 关卡 1：服务端默默挂 SSE 连接（耗时最长）

**现象**：
- DNS / TCP / TLS 全部 OK
- 普通 GET 请求秒级返回（即使 404 也是结构化响应）
- 唯独 SSE 端点（`/sse/connect.do`）：连接建立、请求发送、然后**服务端 0 字节响应直至超时**

**根因**：小米 Aegis 安全平台对 SSE 长连接做"非浏览器客户端"识别，最关键的判定信号是 **TLS 指纹（JA3/JA4）**。Python `requests` / 命令行 `curl` 的 TLS handshake 与 Chrome 不同（加密套件顺序、扩展、椭圆曲线偏好）。

**解法**：
```python
from curl_cffi import requests as cffi_requests
r = cffi_requests.get(url, ..., impersonate="chrome146")
```
`curl_cffi` 内部链接 BoringSSL（Chrome 同款），TLS 指纹与 Chrome 字节级一致，自带 HTTP/2。

**踩过的弯路**：
- 以为是 header 不全 → 把浏览器全套 sec-ch-ua / priority 等头都加上还是不行
- 以为是 HTTP/2 强制 → curl 装不上 HTTP/2 不能直接验证
- 验证手段：先排除一切（DNS/TCP/普通 GET 都正常），定位到"只有 SSE 端点挂"，才确定是协议指纹问题

### 关卡 2：反向 RPC `query.page.json`

**现象**：SSE 收到 `query.page.json` 事件后，AI 处理 20s 后超时报错"获取看板数据超时"。

**根因**：服务端通过 SSE **反向**问浏览器："你现在的页面状态是什么？"——浏览器必须立刻 POST 一个 `save/pageJson.do` 把状态发回去，AI 才能继续。

**解法**：监听 SSE 流，遇到 `query.page.json` 立刻：
```python
POST /api/bigbi/os/agent/save/pageJson.do
body: { "pageJson": "<stringified>", "conversationId": ..., "traceId": ..., "projectId": ... }
```

**关键判定信号**：
- `query.page.json` 事件的 payload 里有 `originTraceId`，必须用这个 traceId 回应（不是 SSE event 的顶层 traceId）
- pageJson 必须是 stringified JSON（不能是对象）
- 即使是空白看板，也要发完整的 menuConfVO + pageSetting 等元信息

### 关卡 3：双层持久化（最隐蔽）

**现象**：所有 endpoint 都返回 200 success，AI 也说"已新增图表"，但**刷新看板没看到任何变化**。

**根因**：BI 系统的"保存"分两层：
- `/agent/page/save.do` — AI 对话内的临时档案，仅用于 AI 撤销/重做语义
- `/page/updatePageDraft.do` — 真正写入看板存储

它们路径前缀不同（`/agent/` vs `/page/`）、payload 结构不同（一个用字符串包装的 pageJson，一个顶层铺平）、含义也不同。

**解法**：在 push.message completed 后，依次调用：
1. `agent/query/chunkData.do` 拉取 AI 生成的 componentList
2. `agent/page/save.do` × 2（before/after，AI 上下文需要）
3. **`page/updatePageDraft.do` 才是真正写盘**

判定写盘成功的信号：response 里 `result` 是个数字字符串（如 `"30"`），代表新版本号。我们的测试看板从 version 0 写到 30——是真实的写入。

### 关卡 4：osMenuId 与 dashboardId 不是一回事

- `dashboardId` = 看板的数据库主键（如 `383540`）
- `osMenuId` = 看板在菜单系统的槽位（如 `dashboard_1`）
- URL `/data-bi/dashboard/383540` 用 dashboardId
- API `?osMenuId=dashboard_1` 用 osMenuId
- 两者不一致，且**不能从 URL 直接推**——必须从 Network 抓任意一个 API 请求的 query 参数里捞

## 三、技术债 / 未解决的问题

| 问题 | 说明 | 后续方向 |
|---|---|---|
| Cookie 过期手动管理 | `_aegis_cas_p` JWT 约 1 天有效，过期后需手动从浏览器重抓 | 长期：用 Playwright 自动登录拿 storage state |

## 五、对接需求文档（给编辑助手 PM）的关键论据

如果要推动对方开放正式 API，今天的发现就是最硬的论据：

1. **TLS 指纹门槛高**：必须模拟 Chrome 完整 TLS 才能调通，普通后端服务器（Python/Java/Go）默认实现都过不了。任何想集成的下游团队都会卡这里。
2. **反向 RPC 设计耦合 UI**：`query.page.json` 这类反向请求假设客户端是浏览器、能立刻提供"页面状态"。后端服务调用没法自然实现这个语义。
3. **持久化双层架构隐蔽**：`/agent/page/save.do` 看名字像"保存"，但实际是 AI 上下文档案；真正写盘要切到 `/page/updatePageDraft.do`。这种设计不写文档没人能猜出来。
4. **多端点协调**：完整建图要 8 个端点协同，部分还需要按时序触发（响应 SSE 事件）。任何下游集成方都会重复踩同样的坑。

**对方提供 SDK 后，下游集成成本能从"几人天逆向 + 持续维护"降到"几小时调用"**——这就是 SDK 的价值。

## 七、本次工程过程的几个值得记住的方法

1. **TLS 指纹假设的验证路径**：DNS → TCP → TLS → 普通 HTTP → 目标 HTTP，逐层排除，最后发现"只有目标端点挂、其他都正常"——指向协议层指纹问题。这是定位非典型网络故障的标准排查链。

2. **HAR 是逆向的最佳武器**：截图会被截断，肉眼看 Network 列表会漏请求，HAR 文件包含毫秒级时间戳 + 完整 payload + 完整 response，配 Python 脚本分析能秒级定位问题。

3. **"成功响应不等于副作用发生"**：服务端返回 200 不一定代表数据真的变了。要看响应体是否包含**变更证据**（如新版本号、生效时间戳等）。本次 `agent/page/save.do` 全部 200 但看板没变就是踩了这个坑。

4. **逆向出来的代码不要进生产**：本次 PoC 完全闭环，但每个环节都是"我们以为是这样"的逆向假设。`whaleVersion: 0`、空白模板的 `version: "0"` 等都是猜测，可能在某些边界场景失败。给对方 PM 看 PoC 也是同样原因——要换正式 API 才能放心。
