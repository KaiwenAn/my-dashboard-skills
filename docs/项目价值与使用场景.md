# dashboard 小助手 · 项目价值与使用场景

> 日期：2026-05-15
> 范围：项目背景 / 价值主张 / 四种使用路径与对应工作场景
> 适用读者：项目作者本人、团队同事、上级评估、潜在协作方

---

## 一、项目背景

### 1.1 你为什么做这件事

数据分析师在日常工作中，**真正消耗精力的不是分析本身，而是把分析做出来的过程**：

- 跟需求方反复对齐"看板该长什么样"
- 写 SQL 反复试错（字段不存在、类型不对、聚合错位）
- 在 BI 平台里手动拖图、配筛选器、调布局
- 中途改需求又要全部返工

这些是**结构化、可重复**的体力活，按理说应该被自动化。但每一步都需要"懂业务 + 懂数据 + 懂工具"才能做对，所以传统的 SQL 模板 / 看板模板都做不彻底。

### 1.2 解决方案的本质

把整个"自然语言需求 → 可用看板"的链路，**拆成 6 个 LLM Agent 串联的流水线**：

```
需求解析 → 语义模型 → BI 推送 → 图表设计 → 指令生成 → 方案生成
```

每个 Agent 只解决一件事，输入输出是结构化 JSON，可独立验证、独立优化。配上**飞书机器人**作为入口，用户在群里 @ 一句需求就能触发整条流水线。

最关键的工程突破在 2026-05-15 完成：**接通了 BI 编辑助手的 API，让最后一步"看板真搭出来"也自动化**。在此之前，用户拿到方案后必须手动复制指令到 BI 平台。现在不用了。

## 二、项目价值

### 2.1 核心价值主张

> **把"想做一个看板"到"看板真做出来"的距离，从多次人工切换压缩到一条飞书消息。**

具体而言：

| 维度 | 之前 | 现在 |
|---|---|---|
| 操作步骤 | 4-5 步：发需求 → 拿方案 → 开 BI 平台 → 新建看板 → 粘贴指令 → 确认 | **1 步**：飞书 @ 机器人 |
| 平均耗时 | 10-20 分钟（含上下文切换） | **3-5 分钟**（人无介入） |
| 注意力消耗 | 跨多个工具，每次切换都要重新进入状态 | 飞书一个窗口，结果自动推回来 |
| 错误兜底 | 失败时从头来一遍 | 失败时仍有方案文档可手动 fallback |
| 流程断点 | 1 个明显断点（人工粘贴指令） | **0 个断点** |

### 2.2 为什么不只是"省时间"

更深层的价值是**让分析师把脑力放在该用的地方**：

- ✅ **应该花时间的事**：理解业务问题、设计指标体系、解读图表、和需求方对话
- ❌ **不应该花时间的事**：拼接 SQL、拖拽图表组件、调整看板布局、记忆 BI 平台的菜单结构

这个工具把后者从工作日中剔除，让分析师每天能多投入 1-2 小时在前者上。

### 2.3 衍生价值

- **沉淀方法论**：每次自然语言需求最后都会生成一份结构化方案文档（md + html），自然成为团队的需求-看板对应关系沉淀
- **降低专业门槛**：业务方也能直接 @ 机器人提需求拿方案，不用再"先找数据分析师排队"
- **可解释性**：从需求文本到 SQL 到看板的每一步都有结构化记录，便于事后追溯（这不是黑盒）

## 三、四种使用路径

### 3.1 路径速查

```
                    要不要把语义模型推到 BI 平台,触发自动建图？
                    ┌─────────────────┴─────────────────┐
                    ↓ 推送                              ↓ 方案（默认）
            自动搭建出真看板                      只产出方案文档
                    │                                    │
            要不要 SQL 试跑校验？                要不要 SQL 试跑校验？
            ┌───┴───┐                            ┌──────┴──────┐
            ↓       ↓                            ↓             ↓
          试跑   不试跑                         试跑         不试跑
            │       │                            │             │
            ↓       ↓                            ↓             ↓
         路径 A  路径 B                       路径 C         路径 D
```

| 路径 | 飞书关键词示例 | 耗时 | 输出 |
|---|---|---|---|
| **A：推送 + 试跑** | "用推送模式做..., 要试跑" | 3-5 min | 方案 + 真看板（数据可靠） |
| **B：推送 + 不试跑** | "用推送模式做..., 不试跑" | 1-2 min | 方案 + 真看板（数据待验证） |
| **C：方案 + 试跑** | "做..., 要试跑" / "仅方案模式..." | 1.5-3 min | 方案文档 + SQL 已校验 |
| **D：方案 + 不试跑** | "做..."（默认） | 30s-1 min | 方案文档（探索性） |

> 详细路径流程与失败终点见 [docs/需求路径全景.md](需求路径全景.md)

### 3.2 路径与工作场景的对应

#### 🅰️ 路径 A：推送 + 试跑（**严谨建图**）

**定位**：当你需要**一次到位、可靠、可直接给人看**的看板。

**典型工作场景**：

| 场景 | 为什么用 A |
|---|---|
| 月度 / 季度业务复盘看板 | 要给业务老板 / 跨部门同事看，**不能有数据错误**。试跑保证 SQL 在数据平台真的能跑通，推送保证看板真做出来。 |
| 给客户 / 外部方做的项目交付看板 | 同上，对外可用性要求高 |
| 上线前的 KPI 监控看板 | 数据正确性是底线 |
| 第一次接触新数据模型 | 自己也不熟悉，需要试跑帮你 catch 错误 |
| 含复杂衍生指标（环比、占比、滚动均值）| 这类 SQL 容易写错，**试跑能挡住 90% 的低级错误** |

**示例**：
> "用推送模式做一个 Q2 销售复盘看板，要试跑。数据源是 dwd_order_detail。要展示按月、按地区的 GMV 和客单价。"

**典型流程**：发需求 → 看到进度卡片走完 6 步（含试跑） → 收到方案 + 链接 → 点链接进 BI 看真看板 → 对内/外发布。

---

#### 🅱️ 路径 B：推送 + 不试跑（**快速建图**）

**定位**：当你**已经熟悉数据**，更想快速把看板搭起来再迭代。

**典型工作场景**：

| 场景 | 为什么用 B |
|---|---|
| 个人**探索性分析** | 你想"先看看数据长什么样"，看板里有没有数据不重要，先把架构搭起来再说 |
| **熟悉的业务+熟悉的数据模型** | 你已经在这数据上写过几十个 SQL，试跑就是浪费时间 |
| 临时需求、demo 用 | 给同事简单展示一下分析思路，不是正式产出 |
| 反复迭代场景 | 第 N 次改同一个看板的某个图，不需要每次都全量试跑 |

**示例**：
> "用推送模式做一个用户行为分析看板, 不试跑。数据源是 dwd_user_module_page_view。"

**风险提示**：图表可能"显示无数据"——SQL 没校验过，字段写错或聚合错位都不会被发现。**适合你能在 BI 平台手动微调 SQL 的场景**，不适合直接给别人用。

---

#### 🅲 路径 C：方案 + 试跑（**严谨方案**）

**定位**：**先与需求方对齐**，对齐后再决定要不要建。这是项目最初**最核心的产品定位**。

**典型工作场景**：

| 场景 | 为什么用 C |
|---|---|
| **新需求初期沟通** | 需求方说"我想看用户行为"，但具体看哪些指标、按什么维度切，并不清楚。先用 C 产出方案文档发给需求方，对齐"指标体系 + 分析思路"再动手搭建 |
| **复杂指标体系设计** | 涉及多张表、复杂衍生指标的项目，要先把指标定义和 SQL 跑通，再决定看板架构 |
| 跨部门协作的看板 | 多个团队对"某指标该怎么算"会有不同意见，方案文档让讨论可视化 |
| 提案 / 讨论 / 评审 | 给老板做"如果建这种看板会展示什么"的提案 |
| **避免"先搭后改"返工** | 搭完了发现指标定义错了，所有图都要重做。先用 C 把方案对齐过，搭出来就是终态 |

**示例**：
> "做一个用户活跃度分析看板，要试跑。数据源是 dwd_user_module_page_view，要看 DAU/WAU、停留时长、关键路径转化率，按部门和用户类型切分。"

**典型流程**：发需求 → 拿到方案 md/html → 转发给需求方 → 收到反馈调整 → **第二次发需求时用 A 模式真建** → 搭出来就是终态。

---

#### 🅳 路径 D：方案 + 不试跑（**快速方案**）

**定位**：纯**探索性思考**，最快产出但最不可靠。

**典型工作场景**：

| 场景 | 为什么用 D |
|---|---|
| **脑暴期** | 自己刚听到需求方提了一句话，先看看 AI 怎么拆解，激发思路 |
| 学习新数据模型 | 想看看"这个表能做出什么样的看板"，不是真要建 |
| **模糊需求的初步成型** | 需求方说"想做个分析",你需要先有一份方案给对方"以这个为起点讨论吗？" |
| 给同事介绍分析思路 | "比如这种数据可以这样切",方案文档自然成了讨论材料 |

**示例**：
> "做一个看板，看用户在工作日和周末的行为差异。"

**风险提示**：方案里的 SQL **可能根本跑不通**（没校验过）——只能当**讨论材料**，不能当工作产出。

---

### 3.3 选择路径的速记口诀

> **看你最关心的那件事**：
>
> - 关心**结果可靠** → A（推送 + 试跑）
> - 关心**速度** → B（推送 + 不试跑）
> - 关心**先对齐再做** → C（方案 + 试跑）
> - 关心**头脑风暴** → D（方案 + 不试跑）

## 四、典型工作日的使用模式

不要把这四条路径当成"互斥的选项"——它们其实**串起来**才能体现最大价值。

### 4.1 推荐组合：D → C → A（探索 → 对齐 → 建图）

**Day 1 上午** —— 接到模糊需求
- 用 **D 路径**对自然语言快速生成方案，看 AI 怎么理解
- 跟需求方初步对话，调整理解

**Day 1 下午** —— 提案对齐
- 改写需求，用 **C 路径**生成正式方案 + 试跑过的 SQL
- 把方案 md/html 发给需求方
- 需求方反馈调整指标和维度

**Day 2** —— 真建
- 整合反馈，用 **A 路径**一键产出真看板
- 直接拿到链接给需求方
- 全流程下来用户**总体只投入对齐 + 验收时间**，约占 30% 的传统耗时

### 4.2 快通道：直接 B（已熟悉的领域）

**自己探索时**：
- 数据熟悉 + 需求清晰 → 直接 **B 路径**快速出看板
- 在 BI 里微调 → 上线
- 单次约 5-10 分钟

### 4.3 严谨通道：直接 A（高 stakes 项目）

**给老板做的复盘 / 给客户的交付物**：
- 直接 A 一次到位
- 跑通就能用，不会有"图表无数据"的尴尬
- 单次约 5-15 分钟（含 LLM + 试跑）

## 五、价值量化（基于本人实测）

### 5.1 时间节省

以一个典型的"用户行为分析看板（10 个图 + 5 个筛选器）"为例：

| 阶段 | 传统方式 | 用本工具（路径 A） | 节省 |
|---|---|---|---|
| 需求理解 + 指标设计 | 30-60 min（与需求方反复 IM） | 5 min（结构化方案文档自带，需求方直接看） | ~80% |
| SQL 编写 + 调试 | 30-90 min | 0 min（LLM 生成 + 试跑保障） | ~100% |
| BI 平台手动建图 | 30-60 min（10 个图拖拽 + 配筛选器） | 1 min（编辑助手自动建图） | ~98% |
| 数据核对 + 微调 | 15-30 min | 15-30 min（不变,这部分仍需人工 review） | 0% |
| **合计** | **105-240 min** | **21-36 min** | **~80%** |

### 5.2 错误率降低

| 错误类型 | 传统方式 | 用本工具 |
|---|---|---|
| SQL 字段名错 / 类型错 | 经常发生（需要逐个图测试） | 试跑 + column_types 注入直接挡住 |
| 聚合 / 窗口函数嵌套错（Spark 严禁） | LLM 生成时常踩,反复修 | prompt v0.7 + 定向修复指引,基本能过 |
| 看板布局拖错 | 偶发（人工拖拽） | LLM 直接生成 layout，一致性好 |
| 指标定义和 SQL 不一致 | 偶发（手工写 SQL 时漏了某个分支） | 方案 md 与 SQL 同源生成,定义一致 |

### 5.3 心智负担降低

最难量化但**主观感受最明显**的价值：

- **不用频繁切换上下文**：飞书一个窗口完成，不用同时开 BI 平台 + IDE + Excel + 飞书
- **不用记忆 BI 平台的操作细节**：菜单结构、组件名字、配置选项都不用想
- **不用怕"漏配某个东西"**：Pipeline 是 deterministic 的，每次都覆盖所有该有的环节

## 六、什么时候不该用（边界）

诚实列出**当前版本的能力上限**，避免误用导致失望：

| 场景 | 为什么不适合 | 怎么办 |
|---|---|---|
| 多用户并发 | 当前 cookie 单账号绑定 | 暂不支持，单用户场景刚好 |
| 跨多个项目（projectId） | `config.json` 里写死 | 改配置后切换；多项目并存需要扩展 |
| 在已有看板上"追加图表" | 当前每次都新建空白看板 | 想加图就在 BI 里手动加，或在新看板里建好后手动迁 |
| 极复杂的 SQL（多 CTE、UDF、子查询嵌套 3 层+）| LLM 生成稳定性会下降 | 用 C 路径产出方案 + SQL 后,自己微调 |
| Cookie 过期时（约 1 天后）| 自动建图会被跳过 | 重抓 cookie 更新 .env，约 30 秒 |
| 数据模型不全 / 字段缺失 | LLM 会"凑合用现有字段"，可能不符合你预期 | 完善上游数据，或用 C 路径先看方案再决定 |

## 七、未来演进方向

按 ROI 排（受益 vs 成本）：

| 优先级 | 方向 | 价值 |
|---|---|---|
| 🟢 P1 | Cookie 自动刷新（Playwright 自动登录拿 storage state） | 消除唯一仍需人工的环节 |
| 🟢 P1 | 推动编辑助手 PM 提供官方 SDK / API | 摆脱内部接口逆向带来的不稳定性 |
| 🟡 P2 | 接已有看板（"在 X 看板上加 Y 图"） | 扩展使用场景 |
| 🟡 P2 | prompt v0.8 整体 audit 去冗余 | v0.7 跑量稳定后做，不破坏载荷 |
| 🟡 P2 | 跨项目支持 | 从 config 写死改成 per-request |
| 🔵 P3 | 团队版（多 cookie 池 / 用户身份隔离） | 让其他同事也能用 |

## 八、一句话总结

> **这是一个把"想做一个看板"和"看板真做出来"之间所有体力活全部自动化的工具。**
>
> 你只需要在飞书发一句自然语言，剩下的 Pipeline 会替你做完——从理解需求、设计指标、生成 SQL、推送到 BI、自动建图，全程透明可追溯。
>
> 它最大的价值不是"省下 X 分钟"，而是**让数据分析师能把脑力花在该花的地方**。
