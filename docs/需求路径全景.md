# 需求路径全景：从飞书消息到看板搭建

> 日期：2026-05-15
> 目标：列清楚用户在飞书发各种需求时，系统会走的完整路径、可能的失败点和最终的反馈消息
> 适用版本：含「编辑助手直连自动建图」集成（[详见](PoC_编辑助手直连_2026-05-15.md)）

---

## 0. 决策树速览

```
用户在飞书 @ 机器人发需求
        │
        ▼
[NLConverter] 解析模式关键词（推送/方案 + 试跑/不试跑）
        │
        ▼
[优先级合并] CLI > 用户 JSON > 关键词 > config.json 兜底
        │
        ▼
   ┌────┴────┐
   ▼         ▼
方案模式      推送模式
(plan)      (publish)
   │         │
   │  ┌──────┴──────┐
   │  │             │
   │  ▼             ▼
   │ 试跑         不试跑
   │  │             │
   │  ▼             ▼
   │ SQL 校验    SQL 不校验
   │ (3 次重试)
   │
   ▼
共同链路: 需求解析 → 语义模型 → BI 推送(只 PUBLISH 走) → 图表设计 → 指令生成 → 方案生成
   │
   ▼
飞书收到方案文档（md + html）
   │
   ▼
[自动建图层] copilot.enabled=true + COPILOT_COOKIE 有效 + 推送模式 + 拿到 model_id ?
   │
   ├─ 否 → 跳过，发说明消息
   └─ 是 → 调编辑助手 9 端点链路，看板真建好 → 发链接
```

## 1. 模式关键词速查

| 用户怎么说 | 触发模式 | 备注 |
|---|---|---|
| "推送…/发布…/publish" | 推送模式 (PUBLISH) | 会把语义模型发到 BI 平台 |
| "不推送…/跳过推送…" | 方案模式 (PLAN) | 只生成方案文档 |
| "仅方案…/只生成方案…" | 方案模式 (PLAN) | 同上 |
| "试跑…/试运行…/校验 SQL…" | 启用试跑 | SQL 在数据平台跑通才往下走 |
| "不试跑…/不校验 SQL…/跳过试跑…" | 关闭试跑 | 直接采用 SQL 不验证 |
| **什么都不说** | 走 [config.json](../scripts/config.json) 默认 | 当前默认：方案 + 不试跑 |

**当前 [config.json](../scripts/config.json) 默认值**：
```json
"bi_platform": { "enabled": "plan" }      // 默认方案模式
"sql_validation": false                    // 默认不试跑
```

如果想"默认就推送"，把 `bi_platform.enabled` 改成 `"publish"`。

## 2. 4 种主路径详解

### 路径 A：推送 + 试跑（最严谨，最慢）

**触发**：飞书消息含"推送"且含"试跑"，例：
> @机器人 推送一个 DAU 趋势看板，数据源是 dm_user_module_behavior，要试跑

**完整流程**：
```
需求解析 → 语义模型(生成SQL) → SQL 试跑(数据平台) → BI 推送 → 图表设计 → 指令生成 → 方案生成 → 自动建图
```

**可能终点**：

| # | 在哪一步出问题 | 用户在飞书收到的消息 | 是否有方案文档 | 是否真建好看板 |
|---|---|---|---|---|
| ① | SQL 试跑 3 次都失败 | "❌ 处理失败：SQL 校验失败：..." | ❌ | ❌ |
| ② | 推送 BI 平台失败（数据源不对/网络/权限） | "❌ 处理失败：BI 推送失败：..." | ❌ | ❌ |
| ③ | 推送成功但下游某步 LLM 异常 | "❌ 处理失败：图表设计失败/..." | ❌ | ❌ |
| ④ | 全部成功 + 自动建图被跳过（cookie 过期/缺失） | 方案文档 + "ℹ️ 自动建图未启用：缺少 COPILOT_COOKIE" | ✅ | ❌ |
| ⑤ | 全部成功 + 自动建图过程中失败（AI 返回 error / updatePageDraft 失败） | 方案文档 + "⚠️ 自动建图未完成：..." | ✅ | ❌（但有失败原因） |
| ⑥ | **全部成功 ✓** | 方案文档 + "✅ 看板已自动搭建完成！🔗 链接" | ✅ | ✅ |

**典型耗时**：2-4 分钟（试跑往往要 30-60s）

---

### 路径 B：推送 + 不试跑（默认推送场景）

**触发**：飞书消息含"推送"但不含"试跑"，例：
> @机器人 推送一个 DAU 趋势看板，数据源是 dm_user_module_behavior

**完整流程**：
```
需求解析 → 语义模型(生成SQL,不校验) → BI 推送 → 图表设计 → 指令生成 → 方案生成 → 自动建图
```

**可能终点**：

| # | 在哪一步出问题 | 用户在飞书收到 | 方案文档 | 看板建好 |
|---|---|---|---|---|
| ① | 推送 BI 平台失败 | "❌ 处理失败：BI 推送失败：..." | ❌ | ❌ |
| ② | 下游某步 LLM 异常 | "❌ 处理失败：..." | ❌ | ❌ |
| ③ | 全部成功 + 自动建图被跳过（无 cookie） | 方案文档 + "ℹ️ 自动建图未启用..." | ✅ | ❌ |
| ④ | 全部成功 + 自动建图过程中失败 | 方案文档 + "⚠️ 自动建图未完成：..." | ✅ | ❌ |
| ⑤ | **全部成功 ✓** | 方案文档 + "✅ 看板已自动搭建完成！🔗 链接" | ✅ | ✅ |

> ⚠️ **不试跑的风险**：SQL 没校验过就推到 BI 平台。看板建出来后，**图表里可能没数据**（SQL 拼错、字段名不对等）。日常推荐还是开试跑。

**典型耗时**：1-2 分钟

---

### 路径 C：方案 + 试跑（保守推荐，先对齐再建）

**触发**：飞书消息含"试跑"但**不含**"推送"（或明确说"不推送"），例：
> @机器人 做一个 DAU 趋势看板，要试跑（先不推送）

**完整流程**：
```
需求解析 → 语义模型(生成SQL) → SQL 试跑 → 图表设计 → 指令生成 → 方案生成
                                                                              ↑
                                                            BI 推送 自动跳过(skipped)
                                                            自动建图 必跳过(无 model_id)
```

**可能终点**：

| # | 在哪一步出问题 | 用户在飞书收到 | 方案文档 | 看板建好 |
|---|---|---|---|---|
| ① | SQL 试跑失败 | "❌ 处理失败：SQL 校验失败：..." | ❌ | ❌ |
| ② | LLM 异常 | "❌ 处理失败：..." | ❌ | ❌ |
| ③ | **全部成功 ✓** | 方案文档 + **"ℹ️ 自动建图跳过：当前是「方案模式」"** | ✅ | ❌（设计如此） |

**为什么自动建图必跳过**：方案模式不调 BI 推送 → 没拿到 `model_id` → 编辑助手 API 不知道用哪个数据模型 → 即使强行调也建不出可用图。

**这个路径的产品价值**：用户拿方案 md/html 给业务方对齐指标和分析思路，**避免"先搭后改"返工**。

**典型耗时**：1.5-3 分钟

---

### 路径 D：方案 + 不试跑（最快，最不严谨）

**触发**：什么都不说（默认） / 明确说"只做方案"，例：
> @机器人 做一个 DAU 趋势看板

**完整流程**：
```
需求解析 → 语义模型(生成SQL,不校验) → 图表设计 → 指令生成 → 方案生成
                                            ↑
                          BI 推送 跳过 + 自动建图 跳过
```

**可能终点**：

| # | 在哪一步出问题 | 用户在飞书收到 | 方案文档 | 看板建好 |
|---|---|---|---|---|
| ① | LLM 异常（需求解析/语义模型/...） | "❌ 处理失败：..." | ❌ | ❌ |
| ② | **全部成功 ✓** | 方案文档 + "ℹ️ 自动建图跳过：当前是「方案模式」" | ✅ | ❌（设计如此） |

> ⚠️ **风险叠加**：SQL 没校验 + 没推到 BI = 方案文档里的 SQL **可能根本跑不通**。最快但最不可靠，仅适合"探索性需求"。

**典型耗时**：30s-1 分钟

## 3. 自动建图叠加层（横切于路径 A、B）

只有路径 A 和 B（推送模式）才有可能触发自动建图。它会在 Pipeline 完成、方案文档已发飞书后才启动。

### 触发条件链（任一条件不满足都跳过）

| 检查 | 不通过时的飞书消息 |
|---|---|
| `config.copilot.enabled = true` | （内部日志，不发飞书）|
| 环境变量 `COPILOT_COOKIE` 非空 | "ℹ️ 自动建图未启用：缺少 COPILOT_COOKIE..." |
| `bi_push_output.skipped = false`（推送成功） | "ℹ️ 自动建图跳过：当前是「方案模式」" |
| 拿到 `model_id` | "ℹ️ 自动建图跳过：bi_push 结果里没拿到 model_id" |
| `instruction_generator_output` 非空 | "ℹ️ 自动建图跳过：instruction_generator 输出为空" |

### 全部条件满足后的内部 9 步

| # | 端点 | 失败时的用户消息 |
|---|---|---|
| 1 | `POST menu/update.do?operationType=create`（建空看板） | "⚠️ 自动建图未完成：创建空白看板失败: ..." |
| 2 | `GET sse/connect.do`（建 SSE 长连） | "⚠️ 自动建图未完成：SSE 失败..." |
| 3 | `POST agent/messages/send.do`（发指令） | "⚠️ 自动建图未完成：send.do 失败..." |
| 4 | `POST agent/save/pageJson.do`（反向 RPC） | （内部，自动重试一次）|
| 5 | 等待 SSE `push.message phase=completed`（最长 180s） | "⚠️ 自动建图未完成：等待 AI 完成超时（180s）" |
| 5b | AI 返回 `interactionType=error` | "⚠️ 自动建图未完成：AI 返回错误：{AI 给出的具体原因}" |
| 6 | `POST agent/query/chunkData.do`（拉成品配置） | "⚠️ 自动建图未完成：chunkData.do 失败..." |
| 7-8 | `POST agent/page/save.do`(before/after)（AI 上下文档案） | （即使失败也继续，不报错） |
| 9 | `POST page/updatePageDraft.do`（真正落库） | "⚠️ 自动建图未完成：updatePageDraft.do 失败..." |
| 全过 | — | "✅ 看板已自动搭建完成！📊 {标题} 🔗 {链接}" |

### 失败降级

**任何自动建图失败都不阻塞主流程**——方案文档（md + html）始终先于自动建图发出。用户至少永远能拿到方案文档，最差就是"自动建图失败，请按方案手动操作"。

## 5. 调试 / 排错速查

| 现象 | 大概率原因 | 怎么定位 |
|---|---|---|
| 一直显示"正在生成方案"无进度 | LLM 服务慢/卡 | 看 `feishu_orchestrator.py` 启动控制台的 `[PROGRESS]` 日志 |
| 收到方案但没"自动建图"消息 | 在方案模式 / cookie 缺失 | 看是否有"ℹ️ 自动建图..."提示消息 |
| "AI 返回错误：xxx" | AI 因为某些字段缺失或语义不匹配放弃 | 把 AI 错误消息复制下来，可能要改 instruction 表述或换数据模型 |
| "updatePageDraft.do 失败" | 通常是 `whaleVersion` / `version` 字段不匹配 | 看 console 里 `body=` 后面的报错 |
| 自动建图成功但点链接看板空白 | 写盘成功但 SQL 跑不出数据 | 在 BI 平台直接编辑那个看板，看图表组件的报错；通常是 `dimensions/metrics` 的字段名和数据模型对不上 |
| Cookie 过期（约 1 天） | `_aegis_cas_p` JWT exp 到了 | 重抓 cookie，按 [PoC 集成文档](PoC_编辑助手直连_2026-05-15.md#使用方法) 步骤更新 .env |

## 6. 一句话决策建议

| 你想做什么 | 推荐路径 |
|---|---|
| 探索性需求，看看 AI 能想到什么 | D（方案 + 不试跑） |
| 准备给业务方对齐方案 | C（方案 + 试跑） |
| 一键产出可用看板（看板本身就是初稿，不需要先对齐） | A（推送 + 试跑）✅ 最稳 |
| 想快速产出，能容忍 SQL 偶尔不靠谱 | B（推送 + 不试跑） |
