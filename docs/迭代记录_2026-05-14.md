## 一、项目总览

### 1.1 是什么

一个飞书 AI 机器人，定位是「**用一句话生成数据看板方案的 AI 助手**」。

业务用户在飞书群里 @ 机器人描述需求（如"做销售看板，数据来自 ods.sales_order"），机器人自动产出可落地的看板方案文档（含可执行 SQL、图表设计、操作指引），可选直接推送到 BI 平台。

### 1.2 核心链路

```
飞书群 @ 机器人
    ↓ WebSocket 长连接（lark-oapi）
[scripts/feishu_orchestrator.py] 飞书入口 + 编排
    ├─ 意图识别（dashboard / revise / unknown）
    ├─ NLConverter（自然语言 → 结构化 JSON + 模式/校验 hint）
    └─ Pipeline（6 步串行执行）
        ① 需求解析       ┐
        ② 语义模型 SQL    ├─ 每步触发进度回调 → 飞书进度卡片
        ③ BI 推送（可选）  │   实时 PATCH 更新（不刷屏）
        ④ 图表设计       │
        ⑤ 看板指令       │
        ⑥ 方案生成       ┘
            ↓
飞书群最终输出：完成通知卡片 + .md 附件 + .html 附件
```

### 1.3 集成的外部系统

| 系统 | 用途 |
|---|---|
| 小米 LLM 网关（`api.llm.mioffice.cn`） | 各 Agent 的 LLM 调用 |
| 小米数据平台 | SQL 试跑 / 表结构查询 |
| 小米 BI 平台 | 看板推送（publish 模式） |
| 飞书开放平台 | WebSocket 长连接事件接收、消息/卡片/文件推送 |

### 1.5 默认配置（[scripts/config.json](scripts/config.json)）

```json
{
  "bi_platform": { "enabled": "plan" },     // 默认仅生成方案，不推送 BI
  "sql_validation": false,                  // 默认不做 SQL 试跑
  "llm": {
    "model": "xiaomi/Qwen3-235B-A22B-Instruct-2507",

```

设计原则：**默认走「保守、安全」档；用户在需求里通过自然语言关键词主动激活「激进、快」档。**

## 二、本次迭代

### 2.1 修复关键 bug

| # | bug | 影响 | 修复 |
|---|---|---|---|
| 1 | `mode="publish"` 写死 | 用户在需求里说"用方案模式"完全无效；config 设 plan 也无效 → **永远推送 BI（不可逆操作）** | 改成 `mode="auto"`，让 hint 和 config 生效 |
| 2 | `_sql_test_hint` 被 pop 后丢弃 | 用户说"不校验 SQL"无效 | 接住 hint 并应用到 `enable_sql_test` |
| 3 | `parse_json_response` 用 `.index()` 抛 `substring not found` | LLM 输出未闭合时 Pipeline 失败，错误信息无意义 | 改用 `.find()` + `-1` 回退分支 |
| 4 | `run_pipeline_in_process` 失败时 result 没传 error 字段 | 用户永远只看到"未知错误" | 透传 `error` + `failed_step` |

### 2.2 稳定性 / 安全性硬伤

| 项 | 改前 | 改后 |
|---|---|---|
| HTTP 超时 | **6 处全部没设** —— 飞书 API 抖动会无限阻塞 | 统一 `HTTP_TIMEOUT=15s`（文件上传单独 60s） |
| HTTP 客户端 | 顶部 import `httpx` 但不用，函数内 6 次重复 import `requests` | 统一顶部 `requests`，删掉 `httpx` |
| Token 缓存 | 多线程并发可能重复请求 | 加 `_token_lock` |
| 任务去重 | `chat_id + text[:20]` 易冲突；无锁 | 改 `chat_id` 维度 + `_tasks_lock` |
| 飞书事件重发 | 同一消息会触发多次生成 | 加 `_processed_message_ids` LRU 幂等去重 |
| 进度卡片刷屏 | 每次重试都 PATCH，可能撞飞书频控 | 节流 `1s`，关键状态 `force=True` 绕过 |
| 默认模型名 | `deepseek-ai/DeepSeek-V4-Flash`（不存在的型号） | `deepseek-chat` + 支持 `LLM_MODEL` 环境变量 |
| 无效配置 | `PIPELINE_TIMEOUT` 配了但代码完全没用到 | 删除 |
| TeeLogger 不关闭 | 进程退出时可能丢失最后日志 | 加 `close()` 方法 + `atexit.register` |

### 2.3 用户体验改进

| 场景 | 改前 | 改后 |
|---|---|---|
| 方案推送 | 长 Markdown 卡片占整屏（截断 3K） + HTML 附件 | 简洁绿色完成卡片（标题/耗时/待确认数）+ **.md 附件** + **.html 附件** |
| 进度卡片标题 | `看板生成中 - 帮我做一个销售看板，数据来自 xxx...`（30 字截断） | `销售看板生成中`（智能提取关键词，无关键词降级到「看板生成中」） |
| 进度条颜色 | `███░░░░░░░ 30%`（黑灰，丑） | `🟦🟦🟦⬜⬜⬜⬜⬜⬜⬜ 30%`（蓝白配色） |
| 意图 unknown | 静默无回复，用户以为机器人挂了 | 引导话术：必填两项 + 完整示例 + "信息越完整越精准" |
| JSON 解析失败 | 静默 | 回复"抱歉，无法解析你的消息内容" |
| @ 标记正则 | 会吃掉正文里的邮箱（`a@b.com`） | 只剥消息开头连续的 @ 标记 |
| Ctrl+C 后卡片 | **永远停留在"正在执行：xxx"中间状态**（飞书侧持久化） | 退出钩子 PATCH 卡片为"服务已中止"+ 发文本提示 |

### 2.4 意图识别完整化

#### 设计目标

`config.json` 提供**默认行为**，用户在需求里通过自然语言**临时覆盖**——这是一个对称、可预测的设计。

#### 三层意图

| 层 | 函数 | 输出 |
|---|---|---|
| 顶层 | `classify_intent` | dashboard / revise / unknown |
| 模式 hint | `_detect_mode_hint` | publish / plan / None |
| SQL 校验 hint | `_detect_sql_test_hint` | True / False / None |

#### 关键改进

| 维度 | 改前 | 改后 |
|---|---|---|
| 模式 hint 实现 | 简单子串匹配；"**不要**推送"被识别为 publish（极危险） | regex + disable 优先；推送对象目标精确化 |
| SQL 校验 hint | 只识别"关闭"关键词，"要校验"识别不了 | 双向覆盖：要 / 需要 / 请 / 麻烦 / 开启 / 启用 + 句首裸命令 + **序列描述**（"语义模型校验**成功后**再推送"） |
| 同义词处理 | "SQL 校验"和"语义模型校验"分开处理 | 在 NLConverter 里视为等价 |
| 跨子句串联 | `glue=.{0,5}` 会让"不要推送，要校验"中的"不要"串到"校验" | `glue=[非标点]{0,5}`，子句界限清晰 |
| 测试覆盖 | 简单几个 case | **67 个用例 + 端到端 10 组合**全部通过 |

#### 端到端效果（前提 `config={plan, false}`）

| 用户在需求里说 | mode 实际 | sql 实际 |
|---|---|---|
| 帮我做个销售看板 | plan | false |
| 帮我做个销售看板，**要推送到 BI** | **publish** | false |
| 帮我做个销售看板，**不要推送** | plan | false |
| 帮我做个销售看板，**要校验语义模型** | plan | **true** |
| 帮我做个销售看板，**不要推送，要校验语义模型** | plan | **true** |
| 帮我做个销售看板，要推送 BI，**不校验** | **publish** | false |
| **语义模型校验成功后再推送 BI** | **publish** | **true** |

每行都符合用户字面意图。

### 2.5 错误信息友好化

新增 `humanize_pipeline_error()`，把 10 类常见错误从"技术堆栈"翻译成"业务建议"，**短摘要给进度卡片**，**完整建议给文本消息**。

| 错误类型 | 短摘要 | 完整建议关键内容 |
|---|---|---|
| 找不到表 | `找不到数据表` | 给出表名 + 检查权限 / 拼写建议 |
| 字段不存在 | `字段不存在` | 给出字段名 + 别名建议 |
| SQL 语法错 | `SQL 语法错误` | 简化需求重试 |
| 类型不匹配 | `字段类型不匹配` | 提示 SUM/AVG 用错或 JOIN 类型不一致 |
| 认证失败 | `数据平台认证失败` | 联系管理员检查 token |
| 连接失败 | `数据平台连接失败` | 稍后重试 |
| 查询超时 | `SQL 执行超时` | 限定时间范围 |
| LLM 限流 | `LLM 调用被限流` | 等几分钟重试 |
| JSON 解析 | `模型输出格式错误` | 多数偶发，重试 |
| **schema 校验** | `模型输出 schema 不规范` | 简化需求 / 联系研发更换模型 |

附加增强：
- 失败时同时显示「失败环节：xxx」，帮用户定位是哪个 Agent 出问题
- 多种错误叠加时短摘要用 `"找不到数据表 等 3 类错误"` 概括
- 未匹配模式时回退到原始错误（截断 400 字）+ 失败环节标注

### 2.6 退出处理（Ctrl+C 友好）

**问题**：飞书消息卡片是服务端持久化的，进程退出后**卡片永远停留在中间状态**，给用户"机器人卡死"的错觉。

**方案**：

1. 全局表 `_active_progress_cards` 记录进行中的卡片
2. `run_pipeline_in_process` 启动时注册、`try/finally` 注销
3. 退出钩子 `_cleanup_active_progress_cards`：
   - PATCH 卡片显示"❌ 服务已中止（Ctrl+C），任务被中断"
   - 发文本消息：`⚠️ 服务已中止，请重启服务后重新发起需求`
4. `atexit.register` + `try/except KeyboardInterrupt` 双保险

### 2.7 模型选择 + 平台限制发现

通过实测探明了一个**重要的平台限制**：

| 路径 | 输出上限 | 说明 |
|---|---|---|
| `siliconflow/Qwen/Qwen3-235B-A22B-Instruct-2507` | **4096 token 硬上限** | 而且 `finish_reason="stop"` 撒谎，不告诉客户端是被截断（导致 `substring not found` 等下游 bug 难以定位） |
| `xiaomi/Qwen3-235B-A22B-Instruct-2507` | 7000+ token，正常自然结束 | ✅ 当前已切换到这条路径 |

→ `config.json` 已指向小米直连版，不再撞 4K 截断。
