## 一、SQL 试跑错误反馈链路修复（最高价值）

### 1.1 问题现象

5月15日的迭代记录里写过一句"试跑 9 次（3 × 3）都不通过——错误反馈被注入了，但 LLM 没把'用 sub-query 拆'转化成正确结构"。当时归因为 LLM 能力不足。

今天复盘发现**不是 LLM 的问题，是反馈链路根本没把真错误传过去**。

### 1.2 排查过程

第一步打开 `poc/diag/sql_fail_attempt*.json` 转储文件：

```json
{
  "failed_models": [{
    "model_name": "dm_user_behavior_overview",
    "error_message": "SQL执行失败",   ← 5 个字符，永远是这个
    "sql": "..."
  }],
  "last_sql_test_error_injected": ""  ← 永远为空（顺便发现的另一个小 bug）
}
```

5 份转储里 `error_message` 全是默认值。

第二步看 [src/data_platform_api.py:273-281](../src/data_platform_api.py#L273-L281) 老代码：

```python
elif state == "ERROR":
    error_code = status_data.get("errorCode", "UNKNOWN")
    error_msg = status_data.get("errorMsg", "SQL执行失败")
    if error_msg == "SQL执行失败":
        error_msg = status_data.get("error", status_data.get("errMsg", error_msg))
```

依次试 `errorMsg` / `error` / `errMsg` 三个字段。但通过加一行 `print(list(status_data.keys()))` 抓到真实响应：

```
['state', 'start', 'completed', 'exceptionMsg', 'progress', 'nextQueryId',
 'logger', 'simpleExceptionMsg']
```

**真实字段是 `exceptionMsg` / `simpleExceptionMsg`**——三个 fallback 候选一个都没命中过。

后续还看到了用户提供的 Kyuubi HTTP API 接入指南文档，第 4 页明确说：

> exceptionMsg：如果 state=ERROR，**显示详细错误信息**

### 1.3 根因影响范围

这个 bug 不光让 LLM 拿不到错误，还让 [src/agents.py:286-299](../src/agents.py#L286-L299) **已经写好的两条「定向修复指引」永远不触发**：

```python
err_lower = sql_test_error.lower()
if "window function" in err_lower and "aggregate function" in err_lower:
    msg += "🔧 定向修复指引：「窗口函数嵌套在聚合函数内」必须用子查询拆开..."
elif "aggregate function in" in err_lower and "another aggregate function" in err_lower:
    msg += "🔧 定向修复指引：「聚合函数嵌套在聚合函数内」必须用子查询拆开..."
```

`err_lower` 永远是 `"sql执行失败".lower()`，关键词永远匹配不上——这两条精心写的修复指引活了好几周但从没运行过一次。

### 1.4 修复

[src/data_platform_api.py:273-291](../src/data_platform_api.py#L273-L291)：

```python
elif state == "ERROR":
    error_code = status_data.get("errorCode", "UNKNOWN")
    raw_msg = (
        status_data.get("simpleExceptionMsg")  # 优先精简版
        or status_data.get("exceptionMsg")     # 完整 Spark 报错
        or status_data.get("errorMsg")
        or status_data.get("error")
        or status_data.get("errMsg")
        or "SQL执行失败"
    )
    # 部分错误带 <div>/<p>/<font> HTML 包装 + Spark logical plan 尾巴
    cleaned = re.sub(r"<[^>]+>", "", raw_msg).strip()
    error_msg = cleaned[:800] + ("..." if len(cleaned) > 800 else "")
    raise SQLExecutionError(error_code, error_msg)
```

为什么 800 字截断：观察到 `exceptionMsg` 里"用户友好层"会附带 Spark 逻辑计划 dump（几千字），对 LLM 反而是噪音；而真正有信息量的部分通常 ≤ 500 字。

### 1.5 验证

修复前后对比，用今天上午两个失败 case 演示：

**Case A（10:11）字段不存在**

修复前 LLM 看到：
```
- dm_user_behavior_daily：SQL执行失败（错误码：UNKNOWN）
```

修复后 LLM 看到：
```
- dm_user_behavior_daily：Error operating EXECUTE_STATEMENT:
  Column 'pa.dept1_name' does not exist. Did you mean one of the following?
  [pv.dept1_name, j.dept1_name, pv.dept2_name, ...]; line 11 pos 42
  （错误码：UNKNOWN）
```

**Case B（同一案例换种写法 10:28）**

整条 pipeline 1 次试跑通过，202 秒走完。这次没失败所以错误注入路径没触发——但代码已上线，下次失败时会按修复后的逻辑工作。

## 二、数据平台 close API 修复

### 2.1 现象

每次 SQL 操作后控制台都会刷一行警告：

```
警告：关闭查询失败（query_id=...）：404 Client Error: Not Found
       for url: https://proxy-service-http-cnbj1-dp.api.xiaomi.net/olap/api/v2/statement/close
```

DESCRIBE 一张表 + 试跑一条 SQL = 至少 4 次 404 刷屏。

### 2.2 根因

之前以为 `/statement/close` 接口不存在，准备加 fallback 静默 404。后来对照 Kyuubi HTTP API 接入指南文档（第 5 页），发现接口存在但**调用格式不对**：

> URL: `{{addr}}/olap/api/v2/statement/close?queryId={{queryId}}`

`queryId` 必须作为 **URL 查询参数**，不是 JSON body。我们老代码把它放进了 body：

```python
url = f"{self.base_url}/olap/api/v2/statement/close"
payload = {"queryId": query_id}
resp = self.session.post(url, json=payload, ...)   # ← 服务器收到没 queryId 的请求，返回 404
```

### 2.3 修复

[src/data_platform_api.py:448-460](../src/data_platform_api.py#L448-L460) 改 `params=` 传 URL 参数：

```python
url = f"{self.base_url}/olap/api/v2/statement/close"
resp = self.session.post(
    url,
    params={"queryId": query_id},  # ← URL 参数
    headers=self._get_headers(),
    timeout=10,
)
```

### 2.4 影响

- 控制台 404 噪音消失
- close 真正生效（虽然平台也会自己回收闲置 query，但显式 close 更礼貌）
- 之后再出现 close 警告就**真的是异常**，可以信任并跟进

文档（第 1 页）特别强调：

> 在 getStatusAndLog 或 fetchResult API 返回 state=ERROR 时，**必须调用 close API 结束本次查询**。

所以这个修复也让 ERROR 路径的资源释放更稳。

## 三、新增 scripts/check_cookie.py

### 3.1 动机

`COPILOT_COOKIE` 有效期约 1 天（`_aegis_cas_p` JWT 的 `exp` 决定）。之前没有快速判断手段——只能等 pipeline 跑到自动建图阶段才知道 cookie 已经挂了，浪费整次 200+ 秒的运行。

### 3.2 实现

[scripts/check_cookie.py](../scripts/check_cookie.py) 通过解码 `_aegis_cas_p` JWT 的 payload 取 `exp` 时间戳，**完全离线、不发任何 HTTP、无副作用**。

关键工程细节：

1. **`latin1` 兜底**：JWT payload 是小米 CAS 生成的，里头 string 字段会塞二进制 escape（如 ``/``），直接 utf-8 decode 在某些 cookie 上会挂 `'utf-8' codec can't decode byte 0xef`。fallback 到 `latin1` 后总能解出来——我们只取 `exp` / `sub` 两个数值/ASCII 字段，不关心字符串内容
2. **Windows GBK 兼容**：`sys.stdout.reconfigure(encoding="utf-8")`，否则 emoji 和中文会在 PowerShell 里报 `UnicodeEncodeError`
3. **换行检测**：用户从 DevTools 拖蓝复制 cookie 容易把视觉换行带成真实换行，pipeline 起跑后会拼出错误的请求头。脚本启动时先检查 `\n` / `\r`，提前给提示

### 3.3 输出含义

| 输出 | 含义 | 行动 |
|---|---|---|
| ✅ Cookie 有效，剩余 X 小时 | 没问题 | 继续干活 |
| ⚠️ 即将过期：剩 X 分钟 | 1 小时内会过期 | 现在就抓新 cookie |
| ❌ Cookie 已过期 | 已经挂了 | 浏览器登 data.mioffice.cn 重抓 |
| ❌ 没找到 _aegis_cas_p | 复制不全 / 来源不对 | 重新抓，整段一行粘进 .env |
| ❌ 中间有换行 | .env 里粘成多行了 | 删掉换行让它合成一行 |

## 四、飞书需求回显扩到 500 字

### 4.1 现象

机器人收到需求后回的"收到！正在生成..."消息会附上"需求：xxx"做回显，方便用户确认 bot 听对了什么。但之前 [scripts/feishu_orchestrator.py:1361](../scripts/feishu_orchestrator.py#L1361) 是硬截断到 100 字符：

```python
reply_text_sync(chat_id, f"收到！...\n\n需求：{text[:100]}\n\n...")
```

用户发"用推送模式做一个用户行为分析看板...数据源是 iceberg_zjyprc_hadoop.meta.dwd_user_module_page_view、..."——后半截就被砍掉了。看不出 bot 是处理了完整需求还是截断后的版本。

### 4.2 修复

[scripts/feishu_orchestrator.py:1361-1362](../scripts/feishu_orchestrator.py#L1361-L1362)：

```python
echo_text = text if len(text) <= 500 else text[:500] + "..."
reply_text_sync(chat_id, f"收到！...\n\n需求：{echo_text}\n\n...")
```

500 字够覆盖绝大多数需求，超出加 `...` 标识让用户知道"被截断了但完整内容已交给 pipeline"。

## 六、已知问题（未修，记录）

下面这些今天没动，留个 backlog：

1. **`Ctrl+C` 后终端不立即返回提示符** ——
   原因 [scripts/feishu_orchestrator.py:1657](../scripts/feishu_orchestrator.py#L1657) 的 `cli.start()`（lark SDK 长连接）内部启了若干非 daemon 线程，`sys.exit(0)` 拦不住它们。临时方案：再按一次 `Ctrl+C`。
   未来修法：把 `sys.exit(0)` 改成 `os._exit(0)`，或者调 lark 的 stop API（如果有的话）

2. **agent 单次执行最慢 ~200 秒** ——
   主要耗在 `semantic_model` 阶段（输入 16K token + 输出 6K token，单次 LLM 调用 ~100s + DESCRIBE 3 张表 ~3s + SQL 试跑 ~5s）。
   优化方向：
   - 缩 prompt（当前 [prompts/semantic-model-agent.md](../prompts/semantic-model-agent.md) 偏厚）
   - bi_push ∥ chart_design 并行（二者都依赖 semantic_model 但互不依赖）
   - 部分 agent 换更小模型（如 instruction_generator 这种结构化输出任务可以用 Haiku 级模型）

3. **`poc/diag/*.json` 转储里 `last_sql_test_error_injected` 永远是空字符串** ——
   原因 [src/agents.py:407-410](../src/agents.py#L407-L410) 的 `finally` 在 super().run() 之后立即把 `_sql_test_error` 从 context 删掉了，而 [src/agents.py:453](../src/agents.py#L453) 的转储读取发生在 finally 之后。要修就把 `last_error` 用局部变量缓存一份再写转储。优先级低，仅影响诊断信息观感

4. **失败 SQL 没有回灌给 LLM** ——
   修完错误注入后 LLM 能看到错误描述，但看不到自己上一次写的失败 SQL。
   对"字段错"/"结构错"影响小（错误描述够），对"微妙逻辑错"（缺 GROUP BY、JOIN 条件错）可能仍乏力。
   等错误注入修复跑一段时间，看真实失败 case 是否仍卡在这种类型，再决定是否需要
